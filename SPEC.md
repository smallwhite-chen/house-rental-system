## 2. 技術堆疊

### 核心技術

| 項目 | 技術 |
|------|------|
| 前端框架 | Next.js 15（App Router） |
| 程式語言 | TypeScript |
| 樣式 | Tailwind CSS |
| ORM | Prisma v7 |
| 資料庫 | PostgreSQL（Supabase 託管） |
| 身份驗證 | NextAuth.js v5 |
| 部署平台 | Vercel |
| 版本控制 | GitHub |

### 補充說明

- **Supabase 連線方式**：Session pooler（非 Direct connection，因 IPv4 相容性問題）
- **NextAuth 登入方式**：支援 Credentials / Google / GitHub 三種登入方式
- **角色設計**：LANDLORD / TENANT 兩種 enum（目前僅實作 LANDLORD，TENANT 預留）
- **設計系統**：Google Material Design 3
- **公司網路限制**：若連線異常，改用手機熱點作為替代方案

### 專案結構（Next.js App Router）

```
/app
  /api              → API Routes（後端邏輯）
  /(auth)           → 登入／登出頁面
  /(dashboard)      → 主系統頁面（需登入）
    /settings       → 系統設定
    /properties     → 房產管理
    /tenants        → 房客管理
    /contracts      → 合約管理
    /finance        → 租金管理
    /communications → 溝通與維修
/components         → 共用元件
/lib                → 工具函式、Prisma Client
/prisma             → Schema 定義
```

## 4. 房產管理（Property Management）

### 4.0 房產類型（PropertyKind）⭐ Phase 10 新增

每個房產建立時必選一種類型，**一經建立不可變更**：

| Kind | 中文 | 例子 | 結構 |
|------|------|------|------|
| `WHOLE_BUILDING` | 整棟型 | 店面、整層出租、獨棟透天 | 房產**直接**為合約主體，無房間概念 |
| `MULTI_UNIT` | 多單位型 | 套房大樓、分租公寓 | 房產底下有多個房間，每個房間獨立簽約 |

#### 建立流程

點「+ 新增房產」→ 跳出 Modal 二選一（整棟型在左，因業務以整棟為主）：

```
🏪 整棟型                       🏢 多單位型
店面、整層出租、獨棟透天          套房大樓、分租公寓

✓ 整棟即合約主體                ✓ 一個房產有多個房間
✓ 不需建立房間單位               ✓ 每個房間獨立簽約
✓ 適合：信義店面、忠孝整層        ✓ 適合：松江大樓 16 間套房
```

選好後再進入對應的房產表單。

#### 用詞通則

系統內統一使用「**單位**」一詞涵蓋房間 + 整棟：
- Dashboard「出租中 N 個單位」
- 列表標題「N 個單位」

#### 內部資料模型

- **MULTI_UNIT**：house → 多個 Unit → 每個 Unit 一份 Contract（現行）
- **WHOLE_BUILDING**：建立時系統**自動產生 1 個隱形 Unit**（number = `"整棟"`），合約掛在此 Unit 上但 UI 不顯示房間管理區

### 4.8 合約建立流程 ⭐ Phase 10 新增

`/contracts/new` 為**單一入口**，依所選房產的 kind 動態調整：

- 選 MULTI_UNIT 房產 → 顯示「關聯房間」下拉（現行行為）

**快捷入口**：兩種房產的詳細頁皆有「+ 新增合約」按鈕，進入時已預填房產。

### 4.9 篩選功能 ⭐ Phase 10 新增

「依房間篩選」（出現於合約列表、支出列表）只列 **MULTI_UNIT 房產旗下的房間**。WHOLE_BUILDING 房產的相關紀錄仍可用「依房產」篩選到。

溝通紀錄
